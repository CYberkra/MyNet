# UavGPR-SimLab UI 重构建议 v0.4

## 结论

当前 GUI 功能完整，但更像“功能页集合”：环境、CSV、生成、队列、后处理、训练入口分散在多个页签中。对熟悉项目的人可用，对新用户不够像成熟仿真软件的“工程向导 + 任务管理器”。

建议不要一次性大重构底层；先做轻量 UI 重构：

1. 首页改为“项目工作台”。
2. 左侧为 5 步流程向导：环境 → 计划 → 预检 → 运行 → 结果。
3. 中间为当前步骤参数。
4. 右侧常驻“任务状态 / 资源 / 最近输出”。
5. 底部常驻日志与错误建议。

## 推荐信息架构

```text
项目工作台
  1. 环境检查
  2. 仿真计划
  3. 预检与去重
  4. 批量运行
  5. 结果与报告
高级设置
  材料参数
  运行计划模板
  HPC/SLURM 模板
  实测 CSV / soft mask
论文工具
  数据集 manifest
  指标汇总
  图表导出
```

## 核心交互优化

### 1. 从“多页签”变成“线性向导 + 可跳转步骤”

用户只需要知道下一步是什么，不需要理解所有模块之间的依赖。

### 2. 运行前必须有“预检页”

预检页显示：

- 将运行多少 case、多少 variant、多少 gprMax 任务。
- 已完成多少、会跳过多少、真正待运行多少。
- GPU/HPC 配置是否满足。
- manifest、job_plan、run_script 路径。

### 3. 队列页突出状态，而不是命令

表格列建议：

```text
状态 | case_id | variant | split | n_traces | 指纹前缀 | GPU | 用时 | 输出 | 操作
```

状态颜色：pending / running / done / skipped / failed。

### 4. 错误信息要可操作

例如：

```text
错误：找不到 conda 环境 gprMax
建议：点击“环境检查”或运行 scripts/OneClick_Install_GPRMAX_Windows.bat
```

### 5. 高级参数折叠

默认只展示：模型数量、标签组件、GPU、运行模式、是否跳过已完成。材料参数、天线、HPC 资源放到高级展开区。

## 是否需要重构

需要，但建议分两级推进：

- v0.4.1：小重构。保留 PySide6 技术栈，重排页面，加入预检/去重/状态表。
- v0.5：中重构。抽象 Project、Job、Artifact 三个核心对象，GUI 全部围绕这三个对象展示。

暂不建议重写为 Web 前端，除非未来要多人远程访问 HPC 队列。
